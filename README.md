We have a primary goal: receive data from the aerosense device, consisting of humidity, wind speed, wind yaw/angle, activity classification, surface classification, CdA, and grade. Note this is separate from the "cadence hack" (ie outputting CdA scaled to fit 0 to 255 in the cadence sensor profile) for sending data out from the aerosense, which wouldn't need a special data field like this.
A secondary goal of boardcasting speed from the Garmin to the device (for when a dedicated ANT+ speed sensor on the wheel is not available). It can use a standard ANT+ broadcast or reuse the existing BLE connection to write speed into the device.
A tertiary goal of being able to set aerosense settings from the Garmin device. For starters users should be able to pass total mass (rider + bike + gear). Also to trigger static pressure calibration via the settings (so from phone to Garmin to aerosense likely).
Coast-down request: a Connect IQ manifest only allows one `<iq:application>`, so a separate "Start Coast Down" data field in the same app isn't possible. Instead the existing full-screen telemetry field handles `onTap` itself, using a two-tap arm/fire sequence. An app setting (`tap_to_coast_enabled`, default on) lets users disable the behaviour. The firmware treats the request as advisory and silently rejects it unless rider power is 0 and the bike is decelerating, so an accidental tap during a hard effort cannot ruin a race.
Pressure-cal request: triggered from the phone-side Garmin Connect settings (`trigger_pressure_cal` oneshot boolean). The data field forwards it on the next compute() tick and clears the flag. The firmware rejects unless the bike is stationary with settled pressure.
Planned: when the data field has enough vertical room (Edge 1040 in a 1-up layout), append a row of native metrics (power, heart rate, distance, lap time) below the aero grid using `Activity.Info`. Deferred until the tap and pressure-cal paths are validated.
Target API level is 5.1.0
